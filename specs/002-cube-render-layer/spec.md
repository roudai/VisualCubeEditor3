# フィーチャー仕様: NxN キューブ 描画レイヤー（sr-visualizer 統合）

**フィーチャーブランチ**: `002-cube-render-layer`
**作成日**: 2026-05-01
**ステータス**: 草稿
**入力**: NxN キューブのコアロジックレイヤー（spec 001）が出力する `CubeState` を SVG / PNG 静止画として描画する純粋な描画アダプターを実装する

## ユーザーシナリオとテスト *（必須）*

### ユーザーストーリー 1 - キューブ状態の SVG 描画（優先度: P1）

開発者（または上位レイヤーの UI/CLI/サーバーサイド処理）が、ロジックレイヤーの
`CubeState` を入力として与え、各ステッカーの配色を反映した SVG 文字列を取得できる。

**この優先度の理由**: V3 が継承すべき V2 のコア価値（高品質なキューブ画像生成）を
最小コストで実現する一次出力経路。Web 埋め込み・印刷・他ツール連携など、
最も汎用性が高いフォーマット。

**独立テスト**: `renderSVG(createCube(3))` が完成状態の 3×3 キューブを
表現する SVG 文字列を返し、SVG 内の rect 要素数が 6面 × 9個 = 54 個であることを
Vitest で検証できる。

**受け入れシナリオ**:

1. **Given** N×N の `CubeState`（N=2〜7）、**When** SVG 描画を要求する、**Then** ステッカー配色が正しく反映された SVG 文字列が返る
2. **Given** 完成状態のキューブ、**When** SVG 描画を要求する、**Then** 6面 × N² 個のステッカー領域が SVG 内に表現される
3. **Given** スクランブル済みのキューブ、**When** SVG 描画を要求する、**Then** 各ステッカーの色が `CubeState` 内の色と一致する
4. **Given** `CubeState`、**When** SVG 描画を実行する、**Then** 元の `CubeState` オブジェクトは変更されない（イミュータブル性の保証）

---

### ユーザーストーリー 2 - キューブ状態の PNG 描画（優先度: P1）

開発者が `CubeState` を入力として与え、ダウンロード可能な PNG バイナリ（バイト配列）を取得できる。

**この優先度の理由**: SVG をサポートしていない外部ツール・ファイル共有用途・
画像投稿サービス向けに、ラスタライズ済みフォーマットが必須。
Web ブラウザ／Node.js 双方で同じ呼び出しから取得できる必要がある。

**独立テスト**: `renderPNG(createCube(3))` が PNG 形式のバイナリを返し、
先頭8バイトが PNG マジックナンバー（`89 50 4E 47 0D 0A 1A 0A`）と一致することを
Vitest で検証できる。

**受け入れシナリオ**:

1. **Given** N×N の `CubeState`、**When** PNG 描画を要求する、**Then** 妥当な PNG バイナリ（マジックナンバー一致）が返る
2. **Given** 同一の `CubeState` と同一の描画オプション、**When** PNG 描画を2回実行する、**Then** 決定論的に同じ内容のバイナリが返る
3. **Given** `CubeState`、**When** PNG 描画を実行する、**Then** 元の `CubeState` オブジェクトは変更されない

---

### ユーザーストーリー 3 - 描画オプションの適用（優先度: P2）

開発者が描画時のオプション（背景色・キューブ寸法・viewport 角度・ステッカー間隔等）を
指定して、見た目をカスタマイズできる。

**この優先度の理由**: 単色背景／透過背景の切替や視点角度の調整は、
利用シーン（ブログ埋め込み・印刷・SNS 共有）ごとに必須。
基本描画（US1/US2）が完成してから上に積む拡張機能として位置付ける。

**独立テスト**: `renderSVG(cube, { backgroundColor: 'transparent' })` が
背景色なしの SVG を返し、`renderSVG(cube, { backgroundColor: '#ffffff' })` との
差分が背景要素の有無のみであることを確認できる。

**受け入れシナリオ**:

1. **Given** `CubeState` と背景色オプション、**When** 描画を要求する、**Then** 指定背景色が反映された出力が返る
2. **Given** `CubeState` と viewport 角度オプション、**When** 描画を要求する、**Then** 指定角度から見たキューブが描画される
3. **Given** オプション未指定（デフォルト値使用）、**When** 描画を要求する、**Then** V2 と同等の標準的な見た目で出力される
4. **Given** 不正なオプション値（範囲外の数値・未定義カラー等）、**When** 描画を要求する、**Then** 明確なエラーが返る（イミュータブル性は維持）

---

### ユーザーストーリー 4 - 標準色スキームの既定値適用（優先度: P2）

オプション指定なしで描画した場合、V2 と同等の標準色スキーム
（Up=白 / Down=黄 / Front=赤 / Back=橙 / Right=青 / Left=緑、または WCA 標準配色）が
既定値として適用される。

**この優先度の理由**: V2 ユーザーの移行体験を損なわないために、
既定の見た目が V2 と一致することは UX 継続性の鍵（Constitution 原則I）。

**独立テスト**: `renderSVG(createCube(3))`（オプション省略）の出力 SVG を
解析し、各面の色が標準色スキームと一致することを Vitest で検証できる。

**受け入れシナリオ**:

1. **Given** 完成状態の `CubeState` とオプション省略、**When** 描画する、**Then** 標準色スキームが適用された画像が返る
2. **Given** カスタム色スキーム（例: Up=ピンク）を指定、**When** 描画する、**Then** 指定色に置き換わった画像が返る

---

### エッジケース

- **N=2（ポケットキューブ）**: スライス層が存在しないが、6面 × 2² = 24 ステッカーが正しく描画される
- **N=7**: 6面 × 7² = 294 ステッカーが正しく描画され、出力サイズが極端に大きくならない
- **空でない手順適用後の状態**: 各ステッカーの色が `CubeState` 内のデータと一致する
- **PNG 出力時の Node.js 環境**: ブラウザ API（Canvas/document）非依存で実行できる
- **連続呼び出し**: 同じ入力に対する出力が決定論的（再現性がある）
- **無効な `CubeState`**: 例えばサイズ整合性が崩れた状態で描画を要求しても、ロジックレイヤー側のバリデーションを前提に最小限の防御で動作する

## 要件 *（必須）*

### 機能要件

- **FR-001**: システムは `CubeState<N>`（N=2〜7）を入力として受け取り、SVG 文字列を出力しなければならない（MUST）
- **FR-002**: システムは `CubeState<N>`（N=2〜7）を入力として受け取り、PNG バイナリ（`Uint8Array`）を出力しなければならない（MUST）
- **FR-003**: 描画レイヤーは `CubeState` を変更してはならない（MUST NOT）。イミュータブル性をロジックレイヤーと同等に保つ
- **FR-004**: 描画レイヤーは UI フレームワーク（Vue 等）・DOM API・ブラウザ API（document/window/Canvas 等）に依存してはならない（MUST NOT）。Node.js 単体で動作しなければならない
- **FR-005**: 描画レイヤーはロジックレイヤー（`src/logic/` の公開 API）の型のみを import 可とする（MUST）
- **FR-006**: 描画オプション（背景色・viewport 角度・ステッカーサイズ等）が省略された場合、V2 と同等の標準色スキーム（白/黄/赤/橙/青/緑）と既定外観を適用しなければならない（MUST）
- **FR-007**: 開発者は描画オプションをパラメータとして渡せなければならない（MUST）。少なくとも背景色・viewport 角度・キューブ全体サイズはカスタマイズ可能とする
- **FR-008**: 描画レイヤーは抽象 `Renderer` インターフェースを定義しなければならない（MUST）。sr-visualizer 実装はその一実装として配置し、将来別実装に差し替え可能とする（Constitution 原則III）
- **FR-009**: 同一入力（`CubeState` + オプション）に対して描画結果は決定論的でなければならない（MUST）。ランダム性・タイムスタンプ・実行環境固有値は出力に含めない
- **FR-010**: 無効な入力（範囲外の N・存在しない色値等）は型付きエラー値として返さなければならない（MUST）。例外を投げない（spec 001 の `Result<T, E>` パターンを踏襲）

### 主要エンティティ

- **Renderer**: 描画レイヤーの抽象インターフェース。`renderSVG(state, options): Result<string, RenderError>` および `renderPNG(state, options): Result<Uint8Array, RenderError>` を提供
- **RenderOptions**: 描画オプション（背景色・viewport 角度・キューブ寸法・色スキーム等）。すべて任意属性で、未指定時は標準値が適用される
- **ColorScheme**: 6面のキューブカラーマッピング（Up/Down/Front/Back/Right/Left → 16進カラー文字列）。既定値は V2 標準スキーム
- **RenderError**: 描画失敗時の型付きエラー。種別（不正サイズ・不正カラー・内部描画ライブラリ失敗等）と人が読めるメッセージを持つ

## 成功基準 *（必須）*

### 測定可能なアウトカム

- **SC-001**: N=2〜7 のすべてのサイズで完成状態のキューブが SVG / PNG として描画され、各ステッカー領域の数が `6 × N²` と一致することが自動テストで確認できる
- **SC-002**: 同一入力に対する描画結果がバイト一致（決定論性）することを自動テストで検証できる
- **SC-003**: 描画レイヤーのラインカバレッジが 90% 以上を達成する（Constitution 原則IV）
- **SC-004**: 描画呼び出しは Node.js 環境で DOM/Canvas を必要とせずに完了する（ヘッドレス Vitest 実行で確認）
- **SC-005**: PNG 出力の先頭8バイトが PNG マジックナンバー（`89 50 4E 47 0D 0A 1A 0A`）と一致する
- **SC-006**: V2 で生成した同一状態の SVG と V3 の SVG を比較したとき、各面の代表色（中央ステッカー色）が完全一致する（Constitution 原則I）
- **SC-007**: すべての公開関数の型定義が TypeScript strict mode でコンパイルエラーなく通過する

## 前提条件

- 描画ライブラリは `sr-visualizer ^1.0.13`（V2 と同一バージョン）を採用する。これ以外の描画ライブラリは本フィーチャーのスコープ外
- アニメーション（GIF/WebP・連続フレーム描画）は本フィーチャーのスコープ外。Constitution 原則I は静止画 PNG と SVG のみ要求する
- 対象キューブサイズはロジックレイヤーと同じく N=2〜7 とする
- 描画レイヤーは `src/render/` 配下に新規作成する純粋なモジュールであり、ストレージ・UI・サーバー等への副作用は持たない
- 色スキームの既定値は V2 標準とし、各面への割り当ては WCA 標準配色（U=白, D=黄, F=赤, B=橙, R=青, L=緑）に揃える
- マスク表示・矢印描画・キューブローテーション（x/y/z）など sr-visualizer の追加機能は、本フィーチャーでは既定無効とし、必要になった時点で別フィーチャーで対応する
- 描画呼び出しは Node.js + ブラウザ双方で同じ呼び出しシグネチャを保証する
