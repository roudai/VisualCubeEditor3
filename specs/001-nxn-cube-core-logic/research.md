# リサーチ: NxN キューブ コアロジックレイヤー

**フィーチャー**: `001-nxn-cube-core-logic`
**作成日**: 2026-04-26

---

## 1. キューブ状態の表現方法

**決定**: `Color` の 2D readonly 配列を6面分持つオブジェクト (`CubeState<N>`)

```
faces[face_index][row][col] = Color
```

**根拠**:
- `face × row × col` の直感的な座標系は可読性が高くデバッグが容易
- `readonly` 修飾により TypeScript がイミュータブル性をコンパイル時に保証できる
- N=2〜7 のサイズでは最大 6×7×7 = 294 要素であり、メモリ効率のための最適化（Uint8Array等）は不要
- `Color` を数値列挙型（0〜5）にすることでシリアライズ/比較が高速

**検討した代替案**:
- `Uint8Array`: 性能は最高だが TypeScript の型安全性が損なわれる。N≤7 では不要
- フラット1次元配列: `face * N² + row * N + col` でアクセスするが可読性が下がる
- Map/Record: 柔軟だが無駄なオーバーヘッドがある

## 2. イミュータブル設計の戦略

**決定**: TypeScript `readonly` 型 + shallow spread コピーによる新状態生成

**根拠**:
- `Object.freeze` は実行時コストがかかり、deep freeze は再帰的でさらに高コスト
- `readonly` + spread は TypeScript が型レベルで変更を禁止し、実行時は通常のオブジェクト操作と同等
- Immer などのライブラリは依存関係を増やし、pure logic layer の方針に反する
- 回転1回あたりの影響範囲は「1面(N×N) + 4スライス辺(4×N)」のみであり、完全コピーのコストは小さい

**検討した代替案**:
- Immer: 使いやすいが、ロジックレイヤーに外部依存を持ち込む。却下
- 完全なディープクローン: シンプルだが性能が線形に悪化する。面配列のみコピーで十分

---

## 3. 回転アルゴリズム

**決定**: サイクル置換（cycle notation）ベースの回転計算

面の回転（N×N グリッド）:
- 時計回り: `result[col][N-1-row] = face[row][col]` の転置
- 反時計回り: `result[N-1-col][row] = face[row][col]`
- 2回転: 時計回りを2回適用

辺・スライスの巡回置換:
- 4つの隣接面から N 個のステッカーを取り出し、回転方向に応じてシフト

**根拠**:
- NxN に対して一般化しやすく、`sliceIndex` パラメータだけで外層〜内層スライスを統一処理できる
- 面数・N のサイズに関係なく同一アルゴリズムが適用できる
- 検証が容易（U → U' で元に戻ることを各テストで確認）

**検討した代替案**:
- 行列・クォータニオン回転: 3D 描画向けであり、ステッカー配列の操作には不要
- ルックアップテーブル: 高速だが N ごとにテーブル生成が必要でメンテナンスコストが高い

## 4. WCA 手順記法のパース

**決定**: 正規表現ベースのトークナイザー + 宣言的マッピングテーブル

WCA 記法の対象範囲:
- 外側面: `U`, `D`, `F`, `B`, `R`, `L`
- 修飾子: `'`（反時計）、`2`（2回転）、なし（時計回り）
- NxN スライス: `2R`, `3U`, `2R'`, `3F2` 等（数字プレフィックス + 面 + 修飾子）
- スコープ外（任意対応）: `x`, `y`, `z`（キューブ全体のローテーション）

トークンの正規表現例:
```
/^(\d+)?([UDFBRL])(w?)([2']?)$/
```

**根拠**:
- 正規表現は WCA 記法の BNF に直接対応でき、実装が小さく保てる
- マッピングテーブル（文字列 → `Move` オブジェクト）により単体テストが容易
- 不正トークンの位置と内容を含むエラーを返しやすい

**検討した代替案**:
- パーサーコンビネーター（Parsec 等）: 表現力は高いが依存関係の追加が必要。却下
- 手書き再帰下降パーサー: 過剰な複雑さ。却下

## 5. キューブ状態の合法性検証

**決定**: 段階的な検証（構造チェック → ステッカー数チェック → パリティチェック）

ステップ:
1. **構造チェック**: faces の次元が `6 × N × N` であることを検証
2. **ステッカー数チェック**: 各色がちょうど N² 個存在することを検証
3. **パリティチェック（3×3のみ完全対応）**: コーナー向き和 mod 3、エッジ向き和 mod 2、置換パリティ

N≥4 では完全な群論的パリティ検証は複雑になるため、ステップ1・2の検証のみを必須とし、
ステップ3は 3×3 に限定して実装する。

**根拠**:
- 保存データ読み込み時の明らかな不正状態（色数異常・次元不正）の検出には1・2で十分
- 完全な合法性検証（N≥4）は研究領域に近く、本プロダクトの要件（SC相当）を超える
- 仕様 FR-007 も「明らかな不正状態を検出できれば十分」と明示している

**検討した代替案**:
- God's Algorithm ベースの完全検証: 性能・実装コストが過大。却下
- 外部ライブラリ（cubing.js 等）の使用: ロジックレイヤーへの外部依存を持ち込む。却下

## 6. エラー処理の方針

**決定**: `Result<T, E>` 型（Either モナドパターン）を使用し、例外を投げない

```typescript
type Result<T, E = LogicError> =
  | { readonly ok: true; readonly value: T }
  | { readonly ok: false; readonly error: E }
```

**根拠**:
- 仕様 FR-009「無効な入力は型付きエラー値として返す」に直接対応
- try/catch なしで呼び出し元が型安全にエラー分岐できる
- TypeScript の型推論と narrowing が自然に機能する
- テストで「エラーが返ること」を例外なしに検証できる

**検討した代替案**:
- `throw Error`: 呼び出し元がエラー処理を強制されず、ランタイムクラッシュのリスク。却下
- `null | undefined`: エラー内容が失われる。却下
