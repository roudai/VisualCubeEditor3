# フィーチャー仕様: NxN キューブ コアロジックレイヤー

**フィーチャーブランチ**: `001-nxn-cube-core-logic`
**作成日**: 2026-04-26
**ステータス**: 草稿
**入力**: NxN キューブのコアロジックレイヤー：キューブ状態管理・回転計算・手順解析

## ユーザーシナリオとテスト *（必須）*

### ユーザーストーリー 1 - キューブ状態の初期化と参照（優先度: P1）

開発者（またはアプリケーション）がサイズ N を指定してキューブを作成し、
各ステッカーの色情報を取得できる。

**この優先度の理由**: 他のすべての操作（回転・手順解析・保存）の基盤となる。
状態表現が固まらないと何も始まらない。

**独立テスト**: `createCube(3)` を呼び出して 3×3 の初期状態を生成し、
全ステッカーが正しい初期色を持つことを Vitest で検証できる。

**受け入れシナリオ**:

1. **Given** サイズ N（2 以上の整数）が与えられた場合、**When** `createCube(N)` を呼び出す、**Then** N×N×6面分のステッカーカラー情報を持つ状態オブジェクトが返る
2. **Given** N=3 の完成状態のキューブ、**When** 特定の面・行・列のステッカーカラーを取得する、**Then** その面の標準色（白・黄・赤・橙・青・緑）が返る
3. **Given** 不正な値（N=1、N=0、負数、非整数）が与えられた場合、**When** `createCube(N)` を呼び出す、**Then** 明確なエラーが返る

---

### ユーザーストーリー 2 - 面回転の適用（優先度: P1）

開発者がキューブ状態に対して任意の面回転（U、D、F、B、R、L およびその時計/反時計/2回転）を適用し、
新しいキューブ状態を得られる。

**この優先度の理由**: キューブの核心操作であり、手順解析・アニメーション・
保存機能のすべてが回転操作に依存する。

**独立テスト**: 3×3 キューブに `U` 回転を1回適用後、影響を受けるステッカーが
正しい位置に移動していることを Vitest で検証できる。
`U` → `U'` の連続適用で元の状態に戻ることも確認できる。

**受け入れシナリオ**:

1. **Given** N×N キューブの状態、**When** 有効な回転記号（例: `U`、`R'`、`F2`）を適用する、**Then** 回転後の新しいキューブ状態が返り、元の状態は変更されない（イミュータブル）
2. **Given** N×N キューブの状態、**When** 同じ回転を4回（または正転 + 逆転）適用する、**Then** 元の状態と一致する
3. **Given** N≥4 のキューブ状態、**When** 内側スライス回転（例: `2R`、`3U'`）を適用する、**Then** 対象スライスのステッカーが正しく移動する
4. **Given** 無効な回転記号が与えられた場合、**When** 適用を試みる、**Then** 明確なエラーが返る

---

### ユーザーストーリー 3 - 手順文字列の解析と適用（優先度: P2）

開発者が WCA 記法の手順文字列（例: `"R U R' U'"`）を解析し、
キューブ状態に一括で適用できる。

**この優先度の理由**: スクランブルの入力・手順の再生・保存データの読み込みなど、
多くのユーザー操作が手順文字列に依存する。回転の単体適用（P1）が完成してから実装できる。

**独立テスト**: `"R U R' U' R U R' U' R U R' U' R U R' U' R U R' U' R U R' U'"` を
3×3 の完成状態に適用し、6手セットの繰り返しで元に戻ることを検証できる。

**受け入れシナリオ**:

1. **Given** WCA 標準記法の手順文字列（スペース区切り）、**When** 解析して手順リストに変換する、**Then** 各トークンが対応する回転オブジェクトの配列として返る
2. **Given** 解析済みの手順リスト、**When** キューブ状態に適用する、**Then** 全手順を順に適用した最終状態が返る
3. **Given** 手順文字列、**When** 逆手順（インバース）を計算する、**Then** 元の状態に戻す逆順手順リストが返る
4. **Given** 不正なトークンを含む文字列、**When** 解析する、**Then** 不正なトークンの位置と内容を示すエラーが返る

---

### ユーザーストーリー 4 - キューブ状態の合法性検証（優先度: P3）

開発者がキューブ状態の合法性（実際に達成可能な状態か）を検証できる。

**この優先度の理由**: 保存データのロード時・外部入力の受け付け時に
不正状態の混入を防ぐために必要。コアロジックの最後に実装する。

**独立テスト**: 完成状態・有効なスクランブル後の状態では合法と判定され、
コーナー1個だけ捻った不正状態では非合法と判定されることを Vitest で検証できる。

**受け入れシナリオ**:

1. **Given** 完成状態のキューブ、**When** 合法性を検証する、**Then** 合法と判定される
2. **Given** 有効な手順を適用して生成したキューブ状態、**When** 合法性を検証する、**Then** 合法と判定される
3. **Given** コーナーやエッジのパリティが不正なキューブ状態、**When** 合法性を検証する、**Then** 非合法と判定される
4. **Given** ステッカー数が正しくない状態オブジェクト、**When** 合法性を検証する、**Then** 非合法と判定される

---

### エッジケース

- N=2（ポケットキューブ）はスライス回転が存在しないが正常に動作しなければならない
- N=7 の大型キューブでも回転計算が正しく完了しなければならない
- 空の手順文字列 `""` を適用した場合は元の状態がそのまま返る
- 回転適用後に元の状態オブジェクトが書き換わってはならない（イミュータブル性の保証）

## 要件 *（必須）*

### 機能要件

- **FR-001**: システムは任意の整数 N（2 ≤ N ≤ 7）を受け取り、NxN キューブの初期完成状態を生成しなければならない（MUST）
- **FR-002**: キューブ状態は6面 × N×N のステッカーカラーを保持しなければならない（MUST）
- **FR-003**: 面回転操作はイミュータブルでなければならない（MUST）。元の状態オブジェクトを変更せず、新しい状態を返す
- **FR-004**: システムは WCA 標準記法（U/D/F/B/R/L、`'` 修飾子、`2` 修飾子、NxN スライス記法 `2R` 等）をサポートしなければならない（MUST）
- **FR-005**: 手順文字列のパーサーは各トークンを個別の回転オブジェクトに変換しなければならない（MUST）
- **FR-006**: システムは任意の手順に対してインバース（逆手順）を計算できなければならない（MUST）
- **FR-007**: システムはキューブ状態の合法性検証を提供しなければならない（MUST）
- **FR-008**: すべての公開関数は DOM・Canvas・ブラウザ API に依存せず Node.js 単体で動作しなければならない（MUST）
- **FR-009**: 無効な入力（範囲外の N、不正な回転記号等）は型付きエラー値として返さなければならない（MUST）

### 主要エンティティ

- **CubeState**: キューブの全ステッカーカラー情報。サイズ N を含み、イミュータブル
- **Face**: 6面のいずれか（Up / Down / Front / Back / Right / Left）
- **Sticker**: 面・行・列で特定される単一ステッカー。カラー値を持つ
- **Move**: 単一の回転操作。対象面・スライス番号・方向（時計/反時計/2回転）を持つ
- **MoveSequence**: Move の順序付きリスト。手順全体を表す
- **Color**: 6色のいずれか（White / Yellow / Red / Orange / Blue / Green）

## 成功基準 *（必須）*

### 測定可能なアウトカム

- **SC-001**: ロジックレイヤーの Vitest ラインカバレッジが 90% 以上を達成する
- **SC-002**: N=2〜7 の全サイズにおいて、任意の手順を適用後にインバースを適用すると元の完成状態に戻ることが自動テストで確認できる
- **SC-003**: 1万手のランダム手順を N=7 のキューブに適用しても 1 秒以内に完了する（ヘッドレス Node.js 環境での計測）
- **SC-004**: すべての公開関数の型定義が TypeScript strict mode でコンパイルエラーなく通過する

## 前提条件

- 対象キューブサイズは N=2〜7 とする。N=1 および N≥8 は本フィーチャーのスコープ外
- 色の表現は文字列ではなく列挙型（または数値）を使用し、パフォーマンスと型安全性を確保する
- 合法性検証（FR-007）は明らかな不正状態を検出できれば十分とし、完全な群論的最適解は必須としない
- WCA 記法のうちキューブローテーション（`x`/`y`/`z`）は任意対応とし、本フィーチャーでは必須としない
- このロジックレイヤーは描画・UI・保存機能には依存しない純粋なモジュールとして提供される
